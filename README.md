# Entity Resolution для клиентских профилей маркетплейса

Проект решает задачу автоматического поиска и объединения дубликатов клиентских профилей по неполным и зашумленным данным: ФИО, email, телефон, дата рождения, пол, география, технические и поведенческие признаки.

Целевой сценарий — использовать модель Entity Resolution в клиентской базе маркетплейса: профили с высокой вероятностью совпадения объединяются автоматически, профили с промежуточной уверенностью отправляются на ручную проверку.

## Текущий статус решения

Текущая реализация находится в ноутбуке `EDA_and_model.ipynb` и покрывает:

- EDA исходных клиентских профилей;
- агрегацию данных до уровня `profile_id`;
- нормализацию identity-полей;
- анализ duplicate-групп через `entity_id`;
- генерацию positive/negative pairwise-примеров;
- feature engineering для сравнения пар профилей;
- обучение baseline-модели Decision Tree;
- обучение финальной модели CatBoost;
- анализ порогов `auto_merge` / `manual_review` / `do_not_merge`;
- графовую кластеризацию через connected components;
- оценку качества кластеризации на validation.
## Бизнес-цель

Дубликаты клиентских профилей приводят к следующим проблемам:

- история покупок и поведение клиента размазываются между аккаунтами;
- персонализация и маркетинговые кампании теряют точность;
- один клиент может получать несколько бонусных контуров;
- LTV, Retention, RFM и когортная аналитика искажаются;
- появляются дополнительные возможности для фрода.

Решение переводит эту проблему в управляемый пайплайн: модель оценивает вероятность совпадения пары профилей, после чего decision layer принимает одно из трех решений.

| Диапазон score | Решение | Интерпретация |
|---:|---|---|
| `score >= 0.85` | `auto_merge` | автоматическое объединение при высокой уверенности |
| `0.60 <= score < 0.85` | `manual_review` | ручная проверка спорной пары / кластера |
| `score < 0.60` | `do_not_merge` | профили не объединяются |

## Данные

Исходный датасет содержит 68 036 строк и 12 колонок. После агрегации выделено 61 927 уникальных `profile_id` и 53 369 уникальных `entity_id`.

Основные поля:

- `created_at` — дата/время появления записи;
- `first_name`, `last_name` — синтетические ФИО;
- `email` — email профиля;
- `phone` — телефон в разных форматах;
- `birthday` — дата рождения;
- `sex` — пол;
- `non_processing_features` — технические/географические признаки;
- `realtime_features` — JSON-контекст: страна, город, timezone, geoid и др.;
- `fs_features` — поведенческие признаки;
- `profile_id` — идентификатор профиля;
- `entity_id` — ground truth сущность клиента.

`entity_id` используется как разметка: если один `entity_id` содержит несколько `profile_id`, такие профили считаются дублями одного клиента.

## Ключевые выводы EDA

- В duplicate-группы входит 16 631 профиль, или 26,86% всех профилей.
- 8 073 `entity_id` содержат более одного `profile_id`.
- Большинство duplicate-кластеров имеют размер 2, но максимальный true-кластер содержит 116 профилей.
- `birthday`, `last_name`, `phone` и `first_name` имеют высокую долю пропусков.
- Полный email уникален на уровне `profile_id`, поэтому exact-match email не подходит как основной метод поиска дублей.
- Нормализованный телефон является высокоточным сигналом, но его покрытие низкое: около 7,5% профилей.
- Технические и поведенческие признаки существенно повышают качество модели, особенно для sparse-профилей.

## Методология

### 1. Агрегация до profile-level

Исходные строки агрегируются по `profile_id`, поскольку один профиль может встречаться несколько раз. Для identity-полей берется первое непустое значение, для временных признаков — минимум и максимум `created_at`, для списочных признаков — объединение уникальных токенов.

### 2. Нормализация

В ноутбуке реализованы следующие преобразования:

- email приводится к нижнему регистру и разделяется на `email_local` и `email_domain`;
- телефон очищается от нецифровых символов и приводится к единому формату;
- текстовые поля очищаются от лишних пробелов и приводятся к нижнему регистру;
- `realtime_features` парсится из JSON;
- `non_processing_features` и `fs_features` превращаются в множества токенов;
- рассчитывается `identity_completeness_score`.

### 3. Pairwise постановка

Задача сведена к бинарной классификации пар профилей:

- positive pair: два профиля имеют одинаковый `entity_id`;
- negative pair: два профиля имеют разные `entity_id`.

Разбиение данных выполнено по `entity_id`, чтобы исключить leakage между train, validation и test.

### 4. Feature engineering

Для каждой пары профилей формируются признаки сходства:

- контактные: совпадение телефона, email-домена, similarity local-part email;
- ФИО: similarity имени и полного имени;
- демографические: пол, год рождения, конфликт известных значений;
- географические: город, регион, страна, geoid;
- технические: устройство, браузер, ОС, `non_processing_features` Jaccard;
- поведенческие: `fs_features` Jaccard, `behavior_mean_sim`;
- временные: разница дат появления профилей;
- признаки полноты: число exact matches, completeness score.

### 5. Модель матчинга

Были сравнены Decision Tree и CatBoost. Decision Tree дал приемлемое качество, но почти полностью опирался на один признак `behavior_mean_sim`, что делает его хрупким.

Финальная модель — CatBoostClassifier:

- `iterations=300`;
- `learning_rate=0.05`;
- `depth=4`;
- `loss_function='Logloss'`;
- `eval_metric='F1'`;
- `auto_class_weights='Balanced'`.

### 6. Кластеризация

После получения вероятностей совпадения пары с `score >= 0.85` добавляются как ребра в граф. Вершины графа — `profile_id`. Connected components интерпретируются как predicted entity-кластеры.

## Рекомендации по внедрению

- Начать с batch-пилота на свежем срезе клиентской базы.
- Не объединять все найденные пары автоматически: использовать высокий порог для auto-merge и отдельную зону ручной проверки.
- Хранить audit log: версия модели, score, порог, top features, исходные значения полей и решение оператора.
- Ввести rollback-процедуру для ошибочно объединенных профилей.
- Обновлять модель на подтвержденных кейсах из manual review.

## Рекомендации по масштабированию

- Вынести blocking в отдельный слой и контролировать candidate recall.
- Использовать несколько blocking-ключей: normalized phone, email-domain/local-prefix, georegion, device/browser/os, MinHash/LSH для поведенческих токенов.
- Для больших объемов использовать Spark/SQL-движок для генерации кандидатов и batch scoring.

## Мониторинг

Необходимо отслеживать:

- pairwise Precision, Recall, F1 на контрольной разметке;
- cluster metrics: ARI, NMI, cluster splitting, cluster merging;
- долю `auto_merge`, `manual_review`, `do_not_merge`;
- latency батчевой обработки;
- drift входных признаков: null-rate, email-домены, телефоны, регионы, устройства
